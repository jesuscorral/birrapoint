# 0010 - Store generated score-sheet PDFs and the results ZIP as Postgres `bytea`

**Status:** Accepted
**Date:** 2026-07-23

## Context

US10 (Event Closing with Automated Results Dispatch, T072–T078) requires generating one PDF per
beer entry, bundling them into a per-competition ZIP (`GET /competitions/{id}/results/archive`,
`202` while generating then `200` streaming the bytes), and emailing each participant their own
PDFs as attachments. All three steps need the previously-generated bytes to still be available
later — the ZIP endpoint can be called well after generation finishes, and `SendResultEmailHandler`
runs as a separate, later `DispatchJob` than `GeneratePdfsHandler`.

Neither `plan.md` nor `data-model.md` ever specified where these bytes should live. `plan.md`'s
Storage section lists only PostgreSQL (server of record) and IndexedDB (client-side offline cache);
`infra/bicep/`'s "persistent storage" note is scoped to the PostgreSQL container itself. There is no
blob storage service anywhere in the approved stack (no Azure Blob Storage, no Azure Files mount
for the API), and adding one would be a new external dependency needing its own justification
(Principle V) purely to hold a few megabytes of PDFs per competition.

## Decision

Store both artifacts directly in Postgres as `bytea` columns, via two new entities not in the
original data-model.md pass:

- `GeneratedScoreSheet(Id, BeerEntryId, PdfBytes, GeneratedAt)` — one row per beer entry, upserted
  by `BeerEntryId` so a `GeneratePdfs` retry never duplicates rows.
- `ResultsArchive(Id, CompetitionId, ZipBytes, GeneratedAt)` — one row per competition, upserted by
  `CompetitionId`.

`GeneratePdfsHandler` writes the former; `BundleZipHandler` reads them all back to build the ZIP
and writes the latter; `SendResultEmailHandler` and the `GET .../results/archive` endpoint both
read already-persisted bytes, never regenerating anything on demand.

## Consequences

- No new infrastructure dependency — everything stays within the already-approved stack
  (PostgreSQL is still the sole server of record; `System.IO.Compression` and QuestPDF were already
  approved dependencies for exactly this purpose per `research.md` R-05).
- Row size: a single score sheet is a few KB; a full competition's ZIP is at most a few MB at MVP
  entry-count volumes. Comfortably within Postgres `bytea`/TOAST limits; not re-evaluated for a
  hypothetical much larger competition.
- `data-model.md` is amended to include both entities and their relationships (Principle X — schema
  changes travel with their implementation). This is a storage-mechanism elaboration of an
  already-approved requirement (FR-036/FR-040/FR-041 already specified PDF/ZIP/email dispatch; only
  the intermediate persistence layer was left unspecified), not a new product requirement — no
  `spec.md` change needed.
- If a future phase needs to scale well beyond MVP volumes (many large competitions, long-term
  results retention), moving these two tables to real blob storage is a contained migration — both
  entities are internal-only (never exposed directly over REST; the API always streams from
  whichever store holds the bytes), so no contract change would be required at that point either.
